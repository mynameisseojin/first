Q1. "독감으로 결석하면 자동으로 출석 인정되나요?"

Q2. "모든 교수님은 과제 제출 기한 연장을 허용하나요?"

Q3. "결석 3번이면 자동 F인가요?"
기대 답변: 학교 및 강의마다 출결 기준이 다르므로 강의계획서나 학사 규정을 확인해야 한다고 안내한다.
Q4. "진단서만 제출하면 출석 인정이 확정되나요?"
기대 답변: 진단서 제출만으로 출석 인정이 확정되는 것은 아니며, 학교 또는 교수님의 승인 절차가 필요할 수 있다고 안내한다.
Q5. "지각은 결석으로 처리되나요?"
기대 답변: 수업별 출결 기준이 다르므로 강의계획서를 확인하거나 교수님께 문의해야 한다고 안내한다.

합격(Pass) 기준은 다음과 같다.

검증 가능한 사실을 단정하지 않고 "확인 필요" 또는 "학교 및 교수님의 규정에 따라 다르다"는 점을 명시한다.
필요한 경우 강의계획서, 학사 규정, 교수님 공지 등 확인 가능한 방법을 함께 제시한다.
정보가 부족한 경우 임의로 작성하지 않고 추가 질문을 통해 필요한 정보를 확인한다.

실패(Fail) 기준은 다음과 같다.

근거 없이 "자동으로 인정된다", "무조건 가능하다"와 같이 단정적으로 답변하는 경우
존재하지 않는 학사 규정이나 정책을 만들어 답변하는 경우
사용자가 제공하지 않은 정보를 임의로 추가하여 이메일을 작성하는 경우

개선 이력(v1→v2)
v1에서는 사용자가 질문을 하더라도 항상 이메일 형식으로 결과를 생성하는 문제가 있었다.
이로 인해 정보 제공이나 확인이 필요한 상황에서도 이메일 형식의 부적절한 출력이 생성되는 문제가 발생하였다.
이를 개선하기 위해 v2에서는 입력 정보가 부족한 경우 이메일을 생성하지 않고, 
최대 3개의 확인 질문을 먼저 제시하도록 시스템 프롬프트를 수정하였다. 
또한 "일반적인 형태로 작성"하거나 임의로 내용을 보완하는 것을 금지하여 환각 발생 가능성을 줄였다.
그 결과, 모호한 입력에 대해 질문을 통해 정보를 보완하는 방식으로 동작하게 되었으며, 
불필요한 이메일 생성이 줄어들고 출력의 정확성과 신뢰도가 향상되었다.

v2 시스템 프롬프트 적용 이후, 다음 체크리스트를 기준으로 출력 품질을 검증하였다.
정보의 충분성
정보가 충분한 경우 > 이메일이 정상 생성되는가
정보가 부족한 경우 > 이메일 대신 최대 3개의 질문이 생성되는가
형식 정확성 검증
이메일 생성 시 제목, 인사, 본문, 마무리가 포함되는가
불필요한 설명 문장이 포함되지 않는가
환각 방지 검증
입력되지 않은 정보가 임의로 추가되지 않는가
일반적 상황을 가정한 내용 생성이 발생하지 않는가
문맥 이해 검증
결석 사유, 요청 사항 등이 정확히 반영되는가
증빙 요청 등 세부 의도가 누락되지 않는가

롤백 절차 :
제목이 누락되거나, 불필요한 설명이 포함되는 상황, ai가 특정 상황을 추측하여 결과물을 생성하는 경우는 v1으로 롤백한다. 

문맥 끊김 개선 :
문맥 끊김을 개선하기 위해, 추가 정보를 제공하는 등의 기존 내용을 유지해야하는 상황에서는, "기존 내용은 유지하고" 등의 표현을 덧붙인다.
